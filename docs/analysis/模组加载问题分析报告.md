## 二、根因分析：ModManager 源码逻辑

基于解包 `sts2_decompiled20260318` 的 `ModManager.cs` 与 `ModManifest.cs` 分析。

### 2.1 模组发现流程（ReadModsInDirRecursive）

```
mods/ 目录（及子目录）递归扫描
    ↓
遍历每个目录中的 .json 文件
    ↓
ReadModManifest(filename) 解析 JSON
    ↓
若 manifest.id == null → 直接拒绝，返回 null，不加入 _mods 列表
    ↓
否则创建 Mod 对象，加入 _mods
```

- 游戏会扫描 **任意以 `.json` 结尾的文件**，因此 `mod_manifest.json`、`DirectConnectIP.json` 都能被发现。
- 文件名可以是 `mod_manifest.json` 或 `{mod_id}.json`，不影响识别。

### 2.2 必填字段：`id`

```csharp
// ModManager.cs 第 330-334 行
if (modManifest.id == null)
{
    Log.Error("Mod manifest " + filename + " is missing the 'id' field! This is not allowed. The mod will not be loaded.", 2);
    mod = null;  // 直接拒绝
}
```

`id` 为 **必需字段**，缺则直接丢弃，不会加入模组列表。

### 2.3 ModManifest 字段规范（ModManifest.cs）

| JSON 字段 | C# 属性 | 必填 | 说明 |
|-----------|---------|------|------|
| `id` | id | ✅ | 模组唯一标识，用于查找 DLL/PCK |
| `name` | name | - | 显示名称 |
| `author` | author | - | 作者 |
| `description` | description | - | 描述 |
| `version` | version | - | 版本 |
| `has_pck` | hasPck | - | 是否有 PCK（默认 false） |
| `has_dll` | hasDll | - | 是否有 DLL（默认 false） |
| `dependencies` | dependencies | - | 依赖模组 id 列表 |
| `affects_gameplay` | affectsGameplay | - | 是否影响游戏性（默认 true） |

注意：JSON 使用 snake_case（`has_pck`、`has_dll`、`affects_gameplay`），由 `JsonPropertyName` 映射。

### 2.4 DLL / PCK 命名规则（TryLoadMod）

```csharp
// 第 432 行：DLL 路径
string text = Path.Combine(mod.path, modId + ".dll");   // {id}.dll

// 第 454 行：PCK 路径
string text2 = Path.Combine(mod.path, modId + ".pck");  // {id}.pck
```

- DLL 必须命名为：`{id}.dll`
- PCK 必须命名为：`{id}.pck`

## 三、DirectConnectIP vs 其他模组对比

### 3.1 DirectConnectIP（✅ 能加载）

```json
{
  "id": "DirectConnectIP",           // ✅ 有 id
  "name": "DirectConnectIP",
  "version": "1.1.3",
  "author": "Tasty_Steak",
  "description": "...",
  "dependencies": [],
  "has_pck": true,                   // ✅ 明确声明 PCK
  "has_dll": true,                   // ✅ 明确声明 DLL
  "affects_gameplay": true
}
```

- 文件结构：`DirectConnectIP.json` + `DirectConnectIP.dll` + `DirectConnectIP.pck`
- 符合游戏要求的 manifest 与 naming convention。

### 3.2 其他模组（ControlPanel、ModConfig、RichPing 等）❌

```json
{
  "pck_name": "ControlPanel",        // ❌ 使用了 pck_name，不是 id
  "name": "控制面板 / Dev Control Panel",
  "author": "VC",
  "description": "...",
  "version": "0.2.0"
  // ❌ 没有 id
  // ❌ 没有 has_pck / has_dll
}
```

- 游戏 **不认识** `pck_name`，只使用 `id`
- 缺少 `id` → 在 `ReadModManifest` 阶段直接返回 `null`，不加入 `_mods`
- 即使补上 `id`，若没有 `has_pck` / `has_dll`，也不会去加载 PCK/DLL

## 四、问题归纳与修复方案

### 4.1 问题 1：缺少 `id` 字段（致命）

- `ModManifest` 中无 `pck_name`，只有 `id`
- 其他模组用 `pck_name` 做标识，未提供 `id`，导致被丢弃。

**修复**：在 manifest 中增加 `id`，建议与 `pck_name` 一致：

```json
"id": "ControlPanel"
```

### 4.2 问题 2：未声明 `has_pck` / `has_dll`

- 若不声明，默认均为 `false`
- 游戏不会去查找 `{id}.pck` / `{id}.dll`，模组资源不会被加载。

**修复**：根据实际需要设置：

```json
"has_pck": true,   // 有 PCK 时
"has_dll": true    // 有 DLL 时
```

### 4.3 问题 3：DLL / PCK 命名与 `id` 不一致

- 游戏只认 `{id}.dll` 和 `{id}.pck`
- 若实际文件名为 `ControlPanel_v2.pck` 等，不会被加载。

**修复**：确保文件名为 `{id}.dll`、`{id}.pck`，且 `id` 与 manifest 中一致。

### 4.4 sts2-heybox-support 的额外问题

- 使用了错误文件名：`mod_mainfest.json`（`mainfest` 应为 `manifest`）
- 即便拼写正确，也需满足上述 manifest 格式要求。

## 五、标准 manifest 模板

供所有模组参考使用的 manifest 模板（参考 DirectConnectIP 格式）：

```json
{
  "id": "YourModId",
  "name": "模组显示名称",
  "version": "1.0.0",
  "author": "作者名",
  "description": "模组描述",
  "dependencies": [],
  "has_pck": true,
  "has_dll": true,
  "affects_gameplay": false
}
```

**目录结构建议**：
```
mods/
  YourModId/
    YourModId.json    ← manifest（文件名任意 .json 均可）
    YourModId.dll     ← 若 has_dll: true
    YourModId.pck     ← 若 has_pck: true
```

## 七、总结

- **DirectConnectIP** 使用的是游戏预期格式，因此能正常加载。
- 其它模组使用的是早期/非官方 manifest（`pck_name`、缺少 `id`、未声明 `has_pck`/`has_dll`），与当前 ModManager 实现不兼容。
- 将 manifest 改为上述标准格式（包含 `id`、`has_pck`、`has_dll` 等），并确保 DLL/PCK 命名为 `{id}.dll`、`{id}.pck`，即可被游戏识别并加载。

## 八、「modid.json」与 manifest 文件名规范（源码核查）

### 8.1 源码中不存在「modid.json」字样

对 `sts2_decompiled20260318` 全文搜索，**未发现** `modid.json` 或类似 `{id}.json` 的硬编码模式。游戏采用通用逻辑：

```csharp
// ModManager.cs ReadModsInDirRecursive（约第 379 行）
foreach (string text in fileIo.GetFilesAt(path))
{
    if (text.EndsWith(".json"))
    {
        Mod mod = ModManager.ReadModManifest(text2, source);
        if (mod != null) ModManager._mods.Add(mod);
    }
}
```

- **扫描规则**：递归遍历 `mods/` 下所有 **以 `.json` 结尾** 的文件
- **文件名**：`mod_manifest.json`、`DirectConnectIP.json`、`{任意}.json` 均可被当作 manifest 尝试解析
- **唯一要求**：JSON 解析成功且 **必须包含 `id` 字段**，否则 `ReadModManifest` 返回 `null`，不会加入模组列表

### 8.2 命名约定建议

| 文件名 | 说明 |
|--------|------|
| `mod_manifest.json` | 通用约定，多数模组使用 |
| `{id}.json` | 如 `DirectConnectIP.json`，与 `id` 一致便于识别 |

两者游戏均支持，**不存在必须用「modid.json」的强制要求**。

## 九、「检测到错误」的根因与解决方向

### 9.1 触发逻辑（NDebugInfoLabelManager.cs）

```csharp
// 第 73-79 行
bool flag = ModManager.LoadedMods.Any((Mod m) => !m.assemblyLoadedSuccessfully.GetValueOrDefault(true));
LocString locString = new LocString("main_menu_ui", "MODDED_WARNING");
locString.Add("count", ModManager.LoadedMods.Count);
locString.Add("hasError", flag);   // hasError=true 时显示红色「检测到错误」
```

- **hasError = true**：当 `LoadedMods` 中**任意模组的 `assemblyLoadedSuccessfully == false`**
- 该标志在 `TryLoadMod` 中设置：`mod.assemblyLoadedSuccessfully = flag4`
- `flag4` 在以下情况被置为 `false`：
  - Harmony `PatchAll` 抛异常（如 NoClientCheats）
  - `CallModInitializer` 抛异常或返回 false（如 sts2-heybox-support）

### 9.2 当前四个问题模组

| 模组 | 错误类型 | 导致 assemblyLoadedSuccessfully=false | 解决方向 |
|------|----------|--------------------------------------|----------|
| **sts2-heybox-support** | `ModManager.GetModNameList Method NotFound` | 初始化时异常 | API 已更名为 `GetGameplayRelevantModNameList()` |
| **NoClientCheats** | Harmony `ModListFilterPatch::TargetMethod()` 返回 null | PatchAll 异常 | 需更新 Harmony 目标方法，适配 v0.99 |
| **DamageMeter** | `ModManifest.pckName` 字段不存在 | 运行时异常（检测 ModConfig 时） | 改用 `manifest.id` 或 `manifest.hasPck`，游戏已弃用 `pckName` |
| **ModConfig** | `Sequence contains no elements`、`Disconnect` 不存在连接 | 注入设置标签页时异常 | 使用 `FirstOrDefault`/`Any` 代替 `First()`，Disconnect 前检查连接是否存在 |

### 9.3 ModManifest 实际字段（v0.99 源码）

`ModManifest.cs` 中**没有 `pckName`**，仅有：

- `id`, `name`, `author`, `description`, `version`
- `hasPck`（JSON: `has_pck`）
- `hasDll`（JSON: `has_dll`）
- `dependencies`, `affectsGameplay`（JSON: `affects_gameplay`）

DamageMeter 等模组若引用 `pckName`，需改为 `id` 或 `hasPck`。

### 9.4 可行解决路径

1. **仅改 manifest JSON**：无法消除「检测到错误」，问题在模组 DLL 代码与游戏 API 的兼容性。
2. **临时消除提示**：禁用问题模组（移除或重命名其 manifest，使游戏不再加载），可去除红色提示。
3. **根本解决**：需要各模组作者更新 DLL，适配 v0.99 的 `GetGameplayRelevantModNameList`、`ModManifest` 字段及 Harmony 目标方法。
