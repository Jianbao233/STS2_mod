## 总目标

将当前 `PvpShopEngine` 从“开关店骨架”推进到可联机验证的商店选卡 `v1`：
- 主机权威
- 可刷新
- 可购买
- 可同步
- 可回放日志

## 执行里程碑

### 里程碑 A：配置化落地（实施路线第 2 步）
1. 定义并接入 `EngineConfig / ModeConfig / StrategyConfig`（含版本字段）。
2. 将当前硬编码常量迁移到配置读取（至少标准模式）。
3. 保留默认回退值，避免配置缺失导致崩溃。

**验收标准**
- 不改变玩法前提下，模式/策略版本可从配置读取并打印日志。

**当前状态（2026-04-15）**
- 已完成：`PvpShopEngineConfig / PvpShopModeDefinition / PvpShopStrategyPack` 已落地，支持 schema/rng/version 字段。
- 已完成：支持配置读取 + 默认回退，缺失或解析失败时会 fallback 并打印日志。
- 未完全完成：配置系统已接好，但当前仓库内未见实际 `shopdraft*.json` 配置文件，标准模式仍主要依赖默认配置运行。
- 里程碑判断：**基本完成（80%）**

---

### 里程碑 B：商店核心运行模型
4. 实装 `ShopState`（槽位、刷新计数、锁定位、stateVersion、历史已看卡等）。
5. 扩展引擎接口到最小可用集：`Open / GetView / Refresh / Purchase / Close`。
6. 建立回合内状态推进（每次刷新/购买都递增版本）。

**验收标准**
- 单机可完整跑通：开店 -> 刷新 -> 购买 -> 关店。

**当前状态（2026-04-15）**
- 已完成：`PvpShopRoundState / PvpShopPlayerState / PvpShopOffer / PvpShopViewModel` 已落地。
- 已完成：`TryOpenRound / TryGetView / TryRefresh / TryPurchase / TryCloseRound` 已实现。
- 已完成：刷新、购买都会更新 `playerState.StateVersion` 与 `roundState.StateVersion`。
- 已完成：回合桥接已接入，主机侧会在回合开始开店、回合结算前关店。
- 风险：当前主要验证到代码路径，未见完整单机手工验收记录或专门 UI 验收链路。
- 里程碑判断：**基本完成（85%）**

---

### 里程碑 C：最小可用出卡管线（v1）
7. 实装 `DeckProfile` 最小分析器（攻/技比例、`Draw/Energy/Block` 缺口、费用缺口）。
8. 实装 `OfferPipeline v1`：候选池 -> 评分 -> 抽样 -> 约束。
9. 首批约束落地：
   - 8 槽模板
   - 功能位保障（`Draw/Energy/Block`）
   - 同轮去重 + 近期降重复

**验收标准**
- 每轮稳定生成 8 张且满足基础保障规则。

**当前状态（2026-04-15）**
- 已完成：`PvpShopDeckProfile` 与 `AnalyzeDeck(...)` 已实现，包含攻/技/能倾向、费用曲线、格挡需求等基础分析。
- 已完成：已实现候选评分、加权抽样、同轮去重、近期重复惩罚、8 槽模板出卡。
- 已完成：已实现最小支持位保底，但目前代码里实际保底主要聚焦 `Block` 支持位。
- 未完成：计划要求中的 `Draw / Energy / Block` 三类功能位完整保障尚未全部落地。
- 里程碑判断：**部分完成（65%）**

- 未完成：未见商店专属网络消息、刷新/购买请求通道、authoritative snapshot 广播/接收链路。
