# ModListHider 排查报告：小眼睛屏蔽效果弱于大眼睛

日期：2026-04-13  
项目：`ModListHider (MLH)`  
范围：仅定位问题，不修改代码

## 1. 用户反馈问题

用户反馈在模组管理页中，使用小眼睛（单模组隐藏）时，个别模组屏蔽不稳定；使用大眼睛（Vanilla Mode 全隐藏）后则可稳定隐藏。  
本次确认对象：`Monster Action Predictor`、`MoreCardRewards`（`QuickLink` 已确认无问题）。

## 2. 日志侧关键事实

来自 `K:\杀戮尖塔mod制作\玩家反馈\godot.log`：

1. MLH 初始化正常，UI 注入正常。
2. 两个目标模组均已加载：
   - `MoreCardRewards`：第 105 行  
   - `Monster Action Predictor`：第 109 行
3. 小眼睛注入与点击事件确实发生：
   - 注入 `Monster Action Predictor` 行：第 249 行  
   - `Monster Action Predictor` 切换 hidden=True：第 272/278/320 行  
   - `MoreCardRewards` 切换 hidden=True：第 284/289/294/300/314 行
4. 没有出现过滤命中日志：
   - 未看到 `Filtered ... hidden mod(s)`  
   - 未看到 `InitialGameInfo: filtered ...`  
   - 未看到 `VanillaMode ON: stripping all ...`（该次日志中大眼睛处于 OFF 状态，见第 304 行）

结论：小眼睛“状态切换”发生了，但联机过滤链路没有明显命中证据。

## 3. 代码侧定位结果

### 3.1 小眼睛保存键来源（UI 侧）

`src/UI/ModMenuRowPatch.cs`：

1. 第 86 行：`string? modId = GetModId(row);`
2. 第 137 行：输出 `Row title: ...`
3. 第 138 行：`return titleText;`

即：当前把 **UI 行标题** 当作 `modId` 传给小眼睛逻辑并落库。

### 3.2 联机过滤匹配键来源（过滤侧）

`src/Core/ModListFilterPatch.cs` 第 54 行：  
`__result?.RemoveAll(id => cfg.ShouldStripFromMultiplayerList(id));`

`src/Core/InitialGameInfoFilterPatch.cs` 第 60 行：  
`list.RemoveAll(id => cfg.ShouldStripFromMultiplayerList(id));`

`src/Config/ModListHiderConfig.cs` 第 46 行开始：  
`ShouldStripFromMultiplayerList(string entry)` 按“联机上报 entry”与 `HiddenModIds` 做精确或 `id-version` 规则匹配。

## 4. 现象解释

当“小眼睛写入的键（UI 标题）”与“联机过滤使用的键（联机上报 id）”不一致时，会出现：

1. UI 看起来切换成功（日志里有 `Toggled ... hidden=True`）。
2. 过滤层命中失败（无 `Filtered ...` 日志，实际联机仍可见或不稳定）。

而大眼睛（Vanilla Mode）走的是“直接清空联机模组列表”分支，不依赖单个键匹配，因此表现更稳定。

## 5. 给后续修改 Agent 的边界说明

1. 本报告只给出定位结论，不包含代码变更。
2. 建议后续修复优先统一“UI 选择键”和“联机过滤键”的数据来源（稳定标识优先）。
3. 建议补充诊断日志：打印每次过滤前后的 entry 列表和命中项，便于验证修复。

## 根因（最终结论）

**当前版本“小眼睛”链路使用了不稳定键（UI 行标题）作为隐藏键，而过滤链路使用的是联机上报条目（稳定 id / id-version 语义）；两者键空间不一致，导致小眼睛屏蔽效果整体弱于大眼睛。**
