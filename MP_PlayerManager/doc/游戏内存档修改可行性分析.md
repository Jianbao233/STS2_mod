## 二、三种修改时机的可行性

| 时机 | 运行时修改 | 存档 JSON 修改 | 备注 |
|------|-----------|--------------|------|
| **跑图中（非战斗）** | ✅ 可行 | ⚠️ 需要拦截保存 | 风险最低 |
| **对局中（战斗中）** | ✅ 可行 | ⚠️ 需要拦截保存 | 战斗状态同步复杂 |
| **退出时（Save & Quit）** | ⚠️ 需要拦截序列化 | ✅ 最简单 | 游戏自动序列化，格式保证 |

### 2.1 跑图中运行时修改（推荐）

**入口**：
```csharp
var runState = RunManager.Instance.DebugOnlyGetState();
if (!RunManager.Instance.IsInProgress) return;  // 必须在跑图中
```

**底层实现**：`IReadOnlyList<Player>` 的底层是 `List<Player>`，可直接反射操作：
```csharp
var playersField = typeof(RunState).GetField(
    "_players",
    BindingFlags.NonPublic | BindingFlags.Instance);
var players = playersField.GetValue(runState) as List<Player>;
players.Add(newPlayer);   // 增
players.Remove(target);   // 删
players[index] = newPlayer; // 替换
```

### 2.2 AddPlayerDebug —— 游戏内置添加玩家方法

```csharp
// RunState.AddPlayerDebug(Player player, int index)
// 源码（第 549-578 行）：
public void AddPlayerDebug(Player player, int index)
{
    if (index >= 0)
        this._players.Insert(index, player);  // 插入指定位置
    else
        this._players.Add(player);              // 追加到末尾

    player.InitializeSeed(this.Rng.StringSeed);  // 初始化随机种子

    foreach (CardModel cardModel in player.Deck.Cards)
        cardModel.AfterCreated();               // 触发卡牌创建后回调

    player.RunState = this;                    // 设置 RunState 引用

    foreach (CardModel cardModel2 in player.Deck.Cards)
        this.AddCard(cardModel2, player);      // 注册到 RunState 卡池

    // 记录 MapPointHistory
    this.CurrentMapPointHistoryEntry.PlayerStats.Add(new PlayerMapPointHistoryEntry {
        PlayerId = player.NetId
    });

    // 如果正在跑图中，初始化遗物袋 + 应用难度加成
    if (RunManager.Instance.IsInProgress) {
        player.PopulateRelicGrabBagIfNecessary(this.Rng.UpFront);
        RunManager.Instance.ApplyAscensionEffects(player);
    }
}
```

**重要**：此方法会：
- ✅ 将玩家加入 `_players` 列表
- ✅ 初始化随机种子（`InitializeSeed`）
- ✅ 触发所有卡牌的 `AfterCreated()` 回调
- ✅ 注册到 `RunState` 的卡池（`AddCard`）
- ✅ 记录到 `MapPointHistory`
- ✅ 如果跑图中，调用 `PopulateRelicGrabBagIfNecessary` 和 `ApplyAscensionEffects`

**注意**：`Player.RunState` 属性只允许设置一次（如果已设置会抛异常）。因此：
- 用 `Player.FromSerializable()` 创建的玩家 → `RunState` 尚未设置 → ✅ `AddPlayerDebug` 可正常设置
- 用 `Player.CreateForNewRun()` 创建的玩家 → 已设置 `RunState = NullRunState` → ⚠️ 需要额外处理

### 2.3 存档 JSON 修改（外部工具路径）

对应 `current_run_mp.save` 文件的 `players[]` 数组：

| 操作 | JSON 操作 |
|------|---------|
| 增玩家 | `players.append(newSerializablePlayer)` |
| 删玩家 | `players.pop(index)` 或 `del players[index]` |
| 替换玩家 | `players[index] = newSerializablePlayer` |

**唯一要求**：`SerializablePlayer.NetId` 必须唯一（不与现有玩家冲突）。游戏通过 `NetId` 追踪玩家。

## 三、三大操作详细分析

### 3.1 增玩家（Add Player）

#### 路径 A：运行时（跑图中）—— ✅ 完全可行

**前置条件**：`RunManager.Instance.IsInProgress == true`

**步骤**：
1. 用 `Player.FromSerializable(serializablePlayer)` 从存档数据构造 `Player` 对象
2. 调用 `runState.AddPlayerDebug(newPlayer, -1)` 追加到玩家列表

**SerializablePlayer 构造器**（Mod 可直接使用 `System.Text.Json` 反序列化）：
```csharp
// Player.FromSerializable 内部实现（第 402-410 行）：
public static Player FromSerializable(SerializablePlayer save)
{
    Player player = new Player(
        ModelDb.GetById<CharacterModel>(save.CharacterId),  // 从 ModelDb 查找角色
        save.NetId,
        save.CurrentHp, save.MaxHp, save.MaxEnergy, save.Gold,
        save.MaxPotionSlotCount, save.BaseOrbSlotCount,
        RelicGrabBag.FromSerializable(save.RelicGrabBag),
        UnlockState.FromSerializable(save.UnlockState),
        save.DiscoveredCards.ToList(),
        save.DiscoveredEnemies.ToList(),
        save.DiscoveredEpochs.ToList(),
        save.DiscoveredPotions.ToList(),
        save.DiscoveredRelics.ToList()
    );
    player.PlayerRng = PlayerRngSet.FromSerializable(save.Rng);
    player.PlayerOdds = PlayerOddsSet.FromSerializable(save.Odds, player.PlayerRng);
    player.ExtraFields = ExtraPlayerFields.FromSerializable(save.ExtraFields);
    player.LoadInventory(save);  // 加载卡组/遗物/药水
    return player;
}
```

**新增玩家的 NetId 生成**：
- 新玩家需要一个**全局唯一的 `NetId`**（`ulong`）
- 可以从现有玩家 NetId 范围外选取（多人模式下通常是 Steam ID）
- 也可直接用 `ulong.MaxValue - 1` 等不冲突值

**风险评估**：
- ✅ `AddPlayerDebug` 是游戏内置方法，逻辑完整
- ⚠️ UI 是否显示新玩家——需验证 `NTopBar` / 玩家选择面板是否响应 `Players` 变化
- ⚠️ 如果在战斗外添加，战斗初始化时是否会自动包含新玩家——需验证

#### 路径 B：存档 JSON —— ✅ 完全可行

在 `current_run_mp.save` 的 `players[]` 数组中追加一个新的 `SerializablePlayer` JSON 对象。

### 3.2 删玩家（Remove Player）

#### 路径 A：运行时 —— ⚠️ 可行但需要多步处理

**步骤**：
1. 反射获取 `RunState._players`（`List<Player>`）
2. 找到目标玩家的索引：`runState.GetPlayerSlotIndex(targetPlayer)`
3. 从列表中移除：`players.Remove(targetPlayer)`

**风险和处理**：

| 风险 | 处理 |
|------|------|
| 新玩家在 `MapPointHistory` 中没有记录 | ✅ `AddPlayerDebug` 会添加 `PlayerMapPointHistoryEntry`；删除时游戏不会自动清理历史 |
| 新玩家的卡牌在 `RunState._runPiles`（卡池）中未移除 | 需要调用 `RunState.RemoveCard(cardModel, player)` 清理 |
| 战斗中新玩家正在操作 | 需要先 `CreatureCmd.Kill(player.Creature, true)` 击杀 |
| `LocalContext.NetId` 指向被删除的玩家 | 如果删的是本地玩家，会导致 `LocalContext.GetMe()` 返回 null |
| 删到只剩 0 个玩家 | 跑图状态应该仍然有效（单人模式下只有 1 个玩家） |

**需要额外 Hook 的地方**：
- `RunState._runPiles`（卡牌注册表）：删除玩家后需要将玩家卡牌从跑图卡池移除
- `CombatState._allies`（战斗盟友列表）：如果战斗中删除玩家，需要同时从 `CombatState` 移除

#### 路径 B：存档 JSON —— ⚠️ 基本可行

从 `players[]` 数组中移除对应元素。

**风险**：删除玩家后，`map_point_history` 中该玩家的 `PlayerStats` 记录仍然存在（无害，游戏读取时只关心 `players[]`）。

---

### 3.3 替换玩家（Replace Player）

#### 路径 A：运行时 —— ⚠️ 可行但需处理引用

**步骤**：
1. 获取目标玩家的槽位索引：`runState.GetPlayerSlotIndex(targetPlayer)`
2. 反射替换 `RunState._players[index]` 为新玩家

**需要处理的问题**：

| 问题 | 说明 |
|------|------|
| `Player.RunState` 只能设置一次 | 新玩家 `FromSerializable` 时 `RunState` 尚未设置，✅ 可以 |
| 旧玩家的卡牌注册在 `RunState._runPiles` | 需要先清空旧玩家卡牌，再注册新玩家卡牌 |
| 旧玩家在 `CombatState._allies` 中 | 需要从 `CombatState` 移除旧玩家 `Creature`，添加新玩家 `Creature` |
| `MapPointHistory` 中旧玩家的 `PlayerStats` | 可以保留（历史记录），也可以覆盖 |
| 旧玩家的 `PlayerRng` / `PlayerOdds` | 新玩家已有自己的随机数状态 |

**完整替换流程**：
```csharp
// 1. 反射获取 _players 列表
var players = GetPrivateField<RunState, List<Player>>("_players");

// 2. 找到并移除旧玩家
int index = runState.GetPlayerSlotIndex(targetPlayer);
players.RemoveAt(index);

// 3. 反射获取并清空旧玩家的卡牌注册
var runPiles = GetPrivateField<RunState, CardPile[]>("_runPiles");
foreach (var card in targetPlayer.Deck.Cards)
    RemoveCardFromRunPiles(runPiles, card);

// 4. 从战斗移除旧玩家 Creature（如在战斗中）
if (CombatManager.Instance.IsInProgress)
    await CreatureCmd.Kill(targetPlayer.Creature, true);

// 5. 创建新玩家（FromSerializable）
var newPlayer = Player.FromSerializable(newSerializablePlayer);

// 6. 用 AddPlayerDebug 重新加入（自动处理所有注册）
runState.AddPlayerDebug(newPlayer, index);
```

#### 路径 B：存档 JSON —— ✅ 最简单

直接替换 `players[index]` 元素。下次加载存档时游戏自动恢复完整状态。

## 四、战斗状态同步（多人联机特有）

### 4.1 CombatState 相关结构

```
CombatState
  ├─ _allies: List<Creature>      ← 玩家生物列表
  ├─ _enemies: List<Creature>       ← 敌人生物列表
  ├─ Allies: IReadOnlyList<Creature>  ← 公开只读视图
  ├─ Enemies: IReadOnlyList<Creature>  ← 公开只读视图
  ├─ Players: IReadOnlyList<Player>   ← 从 PlayerCreatures 筛选
  └─ AddPlayer(Player): void         ← 添加玩家到战斗
```

### 4.2 多人模式下的同步器

`RunManager` 中有一系列 Synchronizer，负责多人状态同步：

| 同步器 | 负责内容 |
|--------|---------|
| `CombatStateSynchronizer` | 战斗状态同步 |
| `PlayerChoiceSynchronizer` | 玩家选择同步 |
| `ActionQueueSynchronizer` | 动作队列同步 |
| `FlavorSynchronizer` | 表现层同步 |

**如果 Mod 在多人联机时添加玩家**：
- ✅ `AddPlayerDebug` 会自动注册到 `RunState`，联机同步器会自动感知新玩家
- ⚠️ 但 `PlayerChoiceSynchronizer` 等可能需要额外处理

### 4.3 KaylaMod 的多人兼容性策略

KaylaMod 通过以下方式避免联机冲突：
- **自签名过滤**：`MultiplayerModSyncCompatPatch` 拦截 `ModManager.GetGameplayRelevantModNameList`，将 Mod 从联机 Mod 列表中移除
- **ModelDb Hash 标准化**：入局握手时规范化 `idDatabaseHash`
- **仅在单人/房主端执行 Mod 逻辑**

**MP_PlayerManager 多人模式策略**（建议）：
- 玩家管理功能**仅在单人模式或房主端执行**
- 通过 `NetService.Type == NetGameType.Singleplayer || NetService.Type == NetGameType.Host` 判断
- 客户端不执行任何修改操作

## 七、结论与建议实现方案

### 7.1 总体可行性

| 操作 | 可行性 | 推荐路径 | 备注 |
|------|--------|---------|------|
| 增玩家（跑图中） | ✅ 可行 | `Player.FromSerializable` + `RunState.AddPlayerDebug` | 游戏内置支持 |
| 删玩家（跑图中） | ⚠️ 可行，需多步处理 | 反射 `_players.Remove` + 清理 `CombatState` | 需处理引用残留 |
